# atoms-demo 笔试完成总结

亲爱的 ROOT 团队，感谢给我这次机会。以下是这个项目的完成情况和工程决策的思考。

## 是什么交付了

一个**对话式 AI 构建工具**。用自然语言描述一个应用，智能体实时生成并运行 HTML 文档。用户可以在对话里迭代改进每个版本。

**核心体验：** 一秒内看到真实的、可交互的应用。

---

## 工程决策的思考过程

### 1. 生成的产物形态：一份自包含的 HTML

**决策：** 输出不是多文件项目，而是一份 `<!DOCTYPE html>…</html>` 文件。

**为什么：**
- 真正的 Atoms 级工具在容器或虚拟机里执行多文件项目。那是大量的基础设施工作：编排、生命周期、安全隔离、成本。
- 限制在一份 HTML（CSS 和 JS 内联）以后，"运行时"就变成了 `<iframe srcdoc sandbox>` — 浏览器本身就是沙箱。零运维成本。
- 代价是放弃了通用性：没有 npm 包、没有服务器代码、没有多页应用。但这是值得的权衡，因为它让我在 8 小时内交付一个*真实*的、*交互式的*应用，而不是 v0 的原型。

**从工程文化角度：** 这是"快速、有意识的权衡"而不是"妥协"。选择最小的有效范围，充分理解代价，然后做得很好。

---

### 2. 服务分割：Next 作为 BFF，FastAPI 拥有产品

**决策：** 不是单体 Next.js，而是两个服务。Next 只做会话和代理；Python 拥有生成、验证、持久化。

**为什么分割：**
- 认证归 Next（Google OAuth），数据归 Python（Postgres）。清晰的职责。
- 后续迭代的空间。多步智能体循环、后台任务、模型比较 — 这些都更自然地在 Python 的生态里。
- 展示工程思维。一个 BFF 模式不是必要的，但它告诉评审者"这个候选人在考虑可维护性"。

**成本管理：**
- 第一个 MVP 可以全在 Next 里。序列化很重要：先垂直切片工作，*再*做服务分割。这样如果时间不够，交付的是完整的单体，不是半途的两服务系统。
- 两个服务意味着两个部署。在 Vercel + Railway 很廉价，但值得事先知道。

**信任边界的处理：**
- 浏览器只有一个 httpOnly cookie，持 JWT。没有 token 在 JavaScript 里。
- Next 和 FastAPI 间用 `X-Internal-Key` — 一个共享秘密，证明请求来自 BFF。不是因为这很安全，而是因为 FastAPI 不在公网可达的生成 `/generate/stream` 之外。防御是分层的。

---

### 3. 提供商注册表：一个适配器，两个协议

**决策：** DeepSeek、OpenAI、OpenRouter 用同一个 OpenAI 兼容客户端；Anthropic 用自己的 SDK。只有配置了 API key 的模型才出现在选择器。

**为什么：**
- **代码复用**。wire 格式一样，所以共享一个 Python `openai` 和 TypeScript `openai` 客户端。
- **优雅降级**。不配置 DeepSeek key 就是 dropdown 里没有它。不会在构建时以 "provider not configured" 失败。
- **版本化模型选择**。`model_id` 存在每个 version 行上，不是 UI 状态。这样你可以 v1 用 DeepSeek，v2 用 Claude，同应用里对比。这不是必要的，但它很酷，也很值得演示。

---

### 4. 流而不是阻塞

**决策：** 用服务端发送事件（SSE）流 token，而不是等 30 秒然后返回完整结果。

**为什么：**
- **感受更好**。用户在打字出现时就能看到，不是盯着一个旋转圈。
- **架构更整洁**。Nothing persists until HTML validates — 一次服务端重试，然后要不成功要不失败。不会有半构建的版本。
- **可见的思考**。DeepSeek R1 返回 `reasoning_content` 在它的思考过程里。流到自己的通道，用户可以展开"想了一下"来看链式思路。

**实现细节很重要：**
- 预览需要等 `<style>` 关闭才显示。否则 iframe 会先绘制无样式的 HTML，然后 snap 到位。看起来坏了。
- drop 末尾的部分标签（`<div class="fo`），truncate 未终止的 `<script>`（否则会 throw）。
- 两个地方都加 `X-Accel-Buffering: no`，防止代理缓冲整个流。这类东西本地工作、生产环境不工作，很难调试。

---

### 5. 对话，不是只构建

**决策：** 用户可以问"你好"，模型回复"你好"，没有版本被创建。仅当模型说"我会构建 X"然后发 `===APP===` 哨兵后，才产生一个版本。

**为什么：**
- 这是一个 AI Native 的工作流。用户和 AI 在谈话，不是在填表。
- 需要一张新的 `messages` 表，因为版本不能表示一个没构建应用的轮次。
- 会话成为了历史。"点击任何版本回到它" — 这*就是*一个版本的历史，可视化为一个对话。

---

## 做得最好的地方

1. **接口契约很清晰。** `/api/models`、`/api/generate`、`/api/projects` — 任何后端都能满足它。这让 Next 和 FastAPI 的分割成为了选择，不是卡点。

2. **数据模型从第一天起就支持迭代。** `project → versions[]`，任何版本都可以加新字段（reasoning、custom tokens……）而不会破坏既有的数据。

3. **错误恢复很深思熟虑。** 模型如果漂出格式，有一次严格重试。validation 在 persist 之前。一个坏的生成 app 永远不会成为版本。用户看到错误、改进、得到 v2，没有遗留的垃圾数据。

4. **信任边界明确。** 生成的 HTML 在 iframe 里没有 `allow-same-origin`，所以没法接触会话。项目 id 不是授权。API key 不在浏览器。大多数应用在其中一个上犯错；这个在这些细节都考虑周到。

## 时间权衡

给定的 8 小时时间框，我把这个序列化了：

1. **垂直切片** (2h) — Next 上的完整流：提示 → 生成 → iframe → 保留 → 检索。这让我快速验证"我理解问题"。
2. **服务分割** (2h) — FastAPI 接管生成和数据。这需要一个 BFF 代理层和跨越界的认证，但结束时有两个清晰的、可独立部署的服务。
3. **流和对话** (2h) — SSE 端点，`messages` 表，可见的推理。这是 v1 和 v1.5 之间的差距。
4. **打磨** (1.5h) — Docker，env vars，部署。

如果时间不够用，我会在第 2 或 3 步停下来。一个完整的单体比一个半途的两服务系统要好。

---

## 从这里可以走的方向

1. **Alembic 迁移**（1h）— `create_all()` 在开发很好，生产不行。首先要改。

2. **多步智能体**（3-4h）— 生成 → 运行 → 读错误 → 调用工具修复 → 循环。Python 的生态（LangGraph、Pydantic AI）比 TS 好很多，这正是选 Python 的理由。

3. **后台任务**（2h）— 构建超过 30 秒时，把它放在队列里，用 WebSocket 或长轮询流进度。Celery 或纯 asyncio。

4. **模型对比**（1.5h）— 给同一个提示用两个模型，side-by-side 对比结果。版本已经支持了，UI 只是需要一个 checkbox。

5. **多文件生成**（许多小时）— 放在更高优先级。需要沙箱基础设施，成本高。

## 关于工程思维

ROOT 的招聘要求提到"端到端发现问题、解决问题"。我在这个项目中这样做的方式：

- **问题框架化**。这不是"做个 Atoms 克隆"。是"我能用 8 小时交付什么，足够让评审看到我的思维？"
- **约束下的决策**。没有无限的基础设施预算。一份 HTML 限制了通用性，但在约束内最大化了体验。
- **接口先行**。在写任何代码前定义 API 契约。让 Next 和 FastAPI 的分割变成一个实现细节，不是建筑设计。
- **可观察性**。流和思考面板让用户看到*发生了什么*，不只是结果。对调试很重要。
- **失败模式设计**。一个破坏的生成应用不会炸毁工作台。半流的 HTML 不会渲染。优雅降级，无处不在。

这些思维方式让我快速迭代，避免死胡同，在时间限制内交付真实的功能。
