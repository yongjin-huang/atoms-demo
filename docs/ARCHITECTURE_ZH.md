# atoms-demo — 架构设计

## 一句话理解这个产品

用自然语言描述一个应用，智能体用一份自包含的 HTML 文档实现它。页面在沙箱 iframe 中实时运行。用户可以在此基础上迭代修改，每一版本都被保留，包括是哪个模型生成的。

---

## 一个决策撑起整个架构

**生成的产物是一份自包含的 HTML 文档。**

真正的 Atoms 级产品生成多文件项目，在容器或浏览器虚拟机中执行——大部分工作是基础设施。

我们把智能体的输出限制在一份 HTML 文件（CSS 和 JS 内联）——这样"运行时"就变成了一个带 `sandbox` 属性的 `<iframe srcdoc>`。浏览器本身就是沙箱。没有基础设施要运维。

| | 多文件 + 容器 | 单个 HTML |
|---|---|---|
| 预览 | 容器或 WebContainer | `<iframe srcdoc sandbox>` |
| 基础设施 | 编排、生命周期管理、清理 | 无 |
| 通用性 | npm 包、服务器代码、路由 | 纯 JS 一个页面 |
| 到第一个工作版本的时间 | 数天 | 数小时 |

**代价是真实的，值得坦白说出来：没有 npm 包、没有服务器代码、没有多页面应用。** 这是有意识的权衡，不是遗漏。为了换来一秒内看到一个真实、交互式应用的体验，我们舍弃了通用性。

下面的每个决策都从这一点衍生。

---

## 数据模型

四张认证表（由认证库所有）加两张产品表。

```
┌──────────────┐
│ user         │   Google 身份。每个人一行。
│──────────────│
│ id      (PK) │
│ email        │
│ name         │
│ image        │
└──────┬───────┘
       │ 1
       │
       │ N
┌──────▼─────────────┐
│ projects           │   用户正在构建的一个应用。
│────────────────────│
│ id          (PK)   │
│ user_id     (FK)   │──► user.id      级联删除
│ title              │   取自第一个提示词
│ created_at         │
└──────┬─────────────┘
       │ 1
       │
       │ N
┌──────▼─────────────┐
│ versions           │   一次生成。只追加不修改。
│────────────────────│
│ id          (PK)   │
│ project_id  (FK)   │──► projects.id  级联删除
│ n                  │   1, 2, 3… 项目内递增
│ prompt             │   这一步用户提了什么
│ html               │   生成的文档
│ model_id           │   用的哪个模型，如 "deepseek/deepseek-chat"
│ created_at         │
└────────────────────┘
```

### 为什么这样设计

**`project → versions[]`，从第一天起就支持版本。** 直观的 v1 设计是平坦的：一个提示、一个输出、一行数据。但"改进这个应用"是让产品感觉活跃的功能，后来再想加这个功能会迫使你迁移*加上* 重写 UI。版本设计在前面，迭代就变成一个 `INSERT`。

**`versions` 只追加。** 一次修改永远不会改动之前的行。这正是版本按钮能工作的原因：v1 到 v4 都还在，都还能跑，在它们之间切换是个客户端数组查找，没有服务器往返。便宜到构建，真实到能用。

**`model_id` 存在版本上，不在 UI 状态里。** 哪个模型生成了哪个版本变成了记录的一部分。这样就能做一个大多数提交做不了的演示时刻：v1 用 DeepSeek，v2 用 Claude，同一个应用，在它们之间切换对比。纯 UI 状态的模型设置做不到这个。

**`n` 在项目范围内，不全局。** 版本号对用户有意义（"我的待办应用的 v3"），所以按项目作用域。插入时计算为 `MAX(n) + 1`。

**`title` 是推导的，不是问出来的。** 永远别在用户看到结果前让他们给东西取名。第一个提示词截断后已经是够好的标签——少了一个字段在意图和结果之间。

### 有意不加的东西

- **没有 `deleted_at`。** 演示里什么都不删。
- **版本上没有 `is_current` 标志。** 当前版本就是 `MAX(n)`。标志是冗余状态，会和真实数据不一致。
- **没有分享/可见性列。** 每个项目对它的所有者私密。后来加公开分享只需一个可空的 `share_token` 列——便宜，所以不值得现在就构建。

## 数据流

### A. 登录

```
浏览器              Next.js               Google              数据库
  │                    │                     │                 │
  │  GET /             │                     │                 │
  ├───────────────────►│                     │                 │
  │                    │ auth() → 无会话     │                 │
  │  ◄── 登录页面 ─────┤                     │                 │
  │                    │                     │                 │
  │  "用 Google        │                     │                 │
  │   继续" (表单)     │                     │                 │
  ├───────────────────►│                     │                 │
  │                    │  重定向到 Google    │                 │
  │  ─────────────────────────────────────►                 │
  │                    │                     │                 │
  │  ◄── 授权屏幕 ─────────────────────────┤                 │
  │  点击允许                               │                 │
  ├───────────────────────────────────────►                 │
  │                    │                     │                 │
  │   /api/auth/callback/google?code=…     │                 │
  ├───────────────────►│                     │                 │
  │                    │ 用 code 交换 token  │                 │
  │                    ├────────────────────►│                 │
  │                    │ ◄── id_token ──────┤                 │
  │                    │                     │                 │
  │                    │ 插入用户、创建会话 │                 │
  │                    ├──────────────────────────────────────►│
  │  ◄── 重定向 / + 会话 cookie ─────────────┤                 │
```

会话 cookie 是 `httpOnly`。浏览器永远不会处理一个 token；每个后续请求会自动携带 cookie。

### B. 构建应用（核心循环）

```
浏览器                Next.js API             提供商            数据库
  │                      │                       │                 │
  │ POST /api/generate   │                       │                 │
  │ { prompt, modelId }  │                       │                 │
  ├─────────────────────►│                       │                 │
  │                      │                       │                 │
  │                      │ 1. auth() — 你是谁？  │                 │
  │                      │    无会话 → 401       │                 │
  │                      │                       │                 │
  │                      │ 2. 验证提示、解析 modelId              │
  │                      │    未知模型 → 400     │                 │
  │                      │                       │                 │
  │                      │ 3. 调用模型           │                 │
  │                      ├──────────────────────►│                 │
  │  (旋转圈)            │  ◄── 原始文本 ────────┤                 │
  │                      │                       │                 │
  │                      │ 4. 从回复中提取 HTML  │                 │
  │                      │    不像 HTML？        │                 │
  │                      │      一次严格重试 ───►│                 │
  │                      │      还是不好 → 500   │                 │
  │                      │                       │                 │
  │                      │ 5. INSERT project     │                 │
  │                      │    INSERT version     │                 │
  │                      ├───────────────────────────────────────►│
  │                      │                       │                 │
  │  ◄─ { projectId, version } ──────────────────────────────────│
  │                      │                       │                 │
  │ router.push(/p/:id)  │                       │                 │
  │ 渲染 <iframe srcdoc={html} sandbox>         │                 │
```

第 4 步是关键的那个。我们告诉模型输出原始 HTML 啥也别说，它通常会听——但"通常"不是个在评审时能跑的演示。提取器去掉 markdown 围栏和任何前言，然后验证。一次严格重试能处理掉第一遍漏掉的几乎所有情况。没有这个，应用大概十次构建有一次会坏，正好够在评审时出现。

### C. 修改（同一个端点）

唯一的区别是请求里带了 `projectId`：

```
POST /api/generate { prompt: "加一个深色模式", projectId, modelId }
  │
  ├─ auth() → userId
  ├─ SELECT project WHERE id = :projectId AND user_id = :userId
  │    没找到 → 404          ← URL 里的 projectId 不是授权
  ├─ SELECT version WHERE project_id = :projectId ORDER BY n DESC LIMIT 1
  │    → previous.html, n = previous.n + 1
  ├─ 用 (之前的 HTML + "改进它: <提示>") 调用模型
  ├─ INSERT version (n, html, model_id)      ← 从不 UPDATE
  └─ → { version }
```

一个端点，两个行为，用 `projectId` 的有无来切换。另一种选择——分开的 `/generate` 和 `/revise` 路由——会复制认证检查、提取、重试和插入。所有权检查和"最新版本"查询是唯一真正的分支。

### D. 加载已保存的项目

`/p/:id` 是个服务器组件。它读会话、在一次服务端传递中获取项目*和它的所有版本*，然后作为 props 传给客户端组件。页面到达时应用已经渲染——没有加载闪烁，没有客户端获取瀑布。

## 接口契约

这是接缝。任何满足它的后端都能服务现在的前端。（见 `BACKEND-CHOICE_ZH.md`。）

### `GET /api/models`

现在哪些模型能实际用。选择器是从这个构建的，所以没有配置 API key 的提供商干脆不出现——它选不到，所以在构建时不会失败。

```json
{
  "models": [
    { "id": "deepseek/deepseek-chat", "label": "DeepSeek V3", "provider": "DeepSeek" },
    { "id": "anthropic/claude-sonnet", "label": "Claude Sonnet", "provider": "Anthropic" }
  ],
  "default": "deepseek/deepseek-chat"
}
```

### `POST /api/generate`

```json
// 请求
{ "prompt": "一个番茄工作法计时器，带任务列表", "modelId": "deepseek/deepseek-chat" }
// ...或者，改进时：
{ "prompt": "加一个深色模式", "projectId": "uuid", "modelId": "…" }

// 200
{
  "projectId": "uuid",
  "version": { "id": "uuid", "n": 1, "prompt": "…", "html": "<!DOCTYPE html>…", "modelId": "…" }
}
```

| 状态 | 何时 |
|---|---|
| 400 | 空提示词，或一个不存在的 `modelId` |
| 401 | 无会话 |
| 404 | `projectId` 不存在**或不属于你** — 两种情况返回同样的响应，所以端点不会泄露哪个 |
| 500 | 模型两次都失败，或提供商出错。Body 包含一条消息，UI 会原样显示。 |

### `GET /api/projects`

当前用户的项目，最新的优先。永远不会是别人的。

```json
[{ "id": "uuid", "title": "一个番茄工作法计时器和任务列表", "createdAt": "…" }]
```

### `GET /api/projects/:id`

一个项目和它的所有版本，按升序。不是你的返回 404。

```json
{ "project": { … }, "versions": [ { "n": 1, … }, { "n": 2, … } ] }
```

## 信任边界

**生成的 HTML 是不可信的输入。** 一个模型写的，用户的提示词引导了模型。它在一个带明确 `sandbox` 属性（`allow-scripts allow-forms allow-modals`）*且不带* `allow-same-origin` 的 iframe 里运行——所以这个 frame 得到一个不透明的原点，没法接触到父文档、它的 cookie，或它的会话。这是前端里最关键的一行代码。

系统提示词也禁止网络调用和远程资源。那是质量指导，不是安全控制——模型可以忽略它。`sandbox` 属性才是真正的控制。

**项目 id 在 URL 里不是授权。** 每次对项目的读和写都在服务端重新检查 `user_id` 对着会话。永远别信 URL 里的 id。

**API key 只在服务端。** 浏览器永远看不到提供商的 key。它发一个 `modelId`；服务端把它解析成一个提供商、base URL 和从环境变量取的一个 key。

## 失败模式和会发生什么

| 失败 | 行为 |
|---|---|
| 模型返回散文而不是 HTML | 提取器去掉它；如果还是无效，一次严格重试；然后一条可读的错误，建议换个模型 |
| 模型超时 | 500 带提供商的消息；版本永远不会写入，没有半构建的行 |
| 提供商 key 缺失 | 这个模型永远不出现在选择器里 |
| 两个构建同时发起 | 按钮在 `working` 时禁用；服务器能容忍它（只追加，`MAX(n)+1`），但 UI 会防止它 |
| 生成的应用在运行时抛错 | 它只在 iframe 内部坏。工作台不受影响。用户读错误、修改、得到 v2。 |

最后一行才是整个设计的意义。一个破坏的生成应用是个正常的、可恢复的事件——不是宕机。
